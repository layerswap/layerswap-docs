## Overview

Follow these best practices to create accessible, performant, and visually appealing widget customizations that provide the best user experience.

## Color Selection

### Use Tailwind Shades for Consistency

<Card title="Tailwind Shades Generator" icon="swatchbook" href="https://tailwindshades.app/">
  Generate professional color palettes with mathematically consistent shades from a single base color.
</Card>

**Why Use Tailwind Shades:**
- Creates consistent color progression (100-900)
- Ensures harmonious color relationships
- Provides predictable hover and active states
- Saves time on manual color selection
- Matches industry-standard color systems

**Workflow:**
1. Enter your brand's primary color
2. Generate the full palette
3. Copy the generated values
4. Convert to Layerswap format (remove `rgb()` wrapper)
5. Test in the playground

### Color Format Requirements

<Warning>
  **Critical**: Always use RGB format without the `rgb()` wrapper.
</Warning>

**Correct:**
```typescript
'99, 102, 241'
```

**Incorrect:**
```typescript
'rgb(99, 102, 241)'  // ❌ Don't include rgb()
'#6366f1'            // ❌ Don't use hex
'hsl(239, 84%, 67%)' // ❌ Don't use HSL
```

### Primary vs Secondary Colors

**Primary Colors:**
- Use your brand's main color as base (500 shade)
- Should be bold and distinctive
- Applied to buttons, links, and key actions
- Ensure sufficient contrast with backgrounds

**Secondary Colors:**
- Typically use neutral grays
- Provide foundation for cards and surfaces
- Should be subtle and not compete with primary
- Consider both light and dark theme scenarios

### Status Color Guidelines

**Maintain Semantic Meaning:**
- Red/Pink: Errors, failures, critical alerts
- Yellow/Amber: Warnings, caution, pending states
- Green: Success, completion, positive actions

**Contrast Requirements:**
- Status foreground must be readable on status background
- Minimum 4.5:1 contrast ratio for text
- Test visibility on both primary and secondary surfaces

## Accessibility

### Contrast Ratios

Follow WCAG 2.1 guidelines for color contrast:

**Normal Text (< 18px):**
- Minimum: 4.5:1 contrast ratio
- Enhanced: 7:1 contrast ratio

**Large Text (≥ 18px or ≥ 14px bold):**
- Minimum: 3:1 contrast ratio
- Enhanced: 4.5:1 contrast ratio

**Testing Tools:**
- [WebAIM Contrast Checker](https://webaim.org/resources/contrastchecker/)
- [Contrast Ratio Tool](https://contrast-ratio.com/)
- Browser DevTools Accessibility Inspector

### Text Readability

**Ensure Readable Text:**
```typescript
// Good: High contrast text
primary: {
  '500': '99, 102, 241',
  'text': '255, 255, 255',  // White on indigo
}

// Bad: Low contrast text
primary: {
  '500': '99, 102, 241',
  'text': '150, 155, 245',  // Light indigo on indigo
}
```

**Button Text:**
```typescript
// Use buttonTextColor to override primary text for buttons
buttonTextColor: '255, 255, 255'  // Always ensure high contrast
```

### Color Blindness Considerations

**Don't Rely on Color Alone:**
- Use icons in addition to colors for status
- Provide text labels where appropriate
- Ensure sufficient brightness differences
- Test with color blindness simulators

**Safe Color Combinations:**
- Blue + Orange (instead of Red + Green)
- Purple + Yellow
- Blue + Brown

## Testing Workflow

### Test in Playground First

<Card title="Interactive Playground" icon="flask" href="https://playground.layerswap.io">
  Always test your theme in the playground before implementing in production.
</Card>

**Testing Steps:**
1. Copy your theme configuration to playground
2. Test with different network combinations
3. Try both swap and CEX flows
4. Test error and success states
5. Verify all interactive elements
6. Test on different screen sizes
7. Export working configuration

### Device Testing

**Test on Multiple Devices:**
- Desktop (various screen sizes)
- Tablet (portrait and landscape)
- Mobile (iOS and Android)
- Different browsers (Chrome, Firefox, Safari, Edge)

**Performance Considerations:**
- Test on lower-end devices
- Monitor for layout shifts
- Check animation smoothness
- Verify blur effects don't cause lag

### Background Compatibility

**Test on Your Application Background:**
```typescript
// If your app has a dark background
backgroundColor: '#0a0a0a'

// If your app has a light background
backgroundColor: '#ffffff'

// If your app has a colored background
backgroundColor: '#1e3a8a'
```

Ensure your widget theme works on your actual application background, not just a white or black test background.

## Performance Optimization

### Backdrop Filter Effects

**Use Sparingly:**
```typescript
// Good: Light blur
cardBackgroundStyle: {
  backdropFilter: 'blur(10px)',
}

// Problematic: Heavy blur on low-end devices
cardBackgroundStyle: {
  backdropFilter: 'blur(40px) saturate(200%) contrast(150%)',
}
```

**Best Practices:**
- Keep blur values under 20px
- Limit number of backdrop filters
- Test on mobile devices
- Consider fallback for unsupported browsers

### Custom Gradients

**Optimize Gradient Complexity:**
```typescript
// Good: Simple gradient
background: 'linear-gradient(135deg, rgba(99, 102, 241, 0.1) 0%, rgba(139, 92, 246, 0.1) 100%)'

// Avoid: Complex multi-stop gradients
background: 'linear-gradient(135deg, rgba(99, 102, 241, 0.1) 0%, rgba(139, 92, 246, 0.1) 25%, rgba(236, 72, 153, 0.1) 50%, rgba(234, 179, 8, 0.1) 75%, rgba(34, 197, 94, 0.1) 100%)'
```

### Box Shadows

**Use Moderate Shadows:**
```typescript
// Good: Subtle shadow
boxShadow: '0 10px 40px rgba(0, 0, 0, 0.1)'

// Avoid: Excessive shadows
boxShadow: '0 0 100px 50px rgba(99, 102, 241, 0.8), 0 0 200px 100px rgba(139, 92, 246, 0.6)'
```

## Common Pitfalls to Avoid

### Color Format Errors

**Most Common Mistake:**
```typescript
// ❌ Including rgb() wrapper
primary: {
  DEFAULT: 'rgb(99, 102, 241)',
}

// ✅ Correct format
primary: {
  DEFAULT: '99, 102, 241',
}
```

### Incomplete Color Palettes

**Don't Skip Shades:**
```typescript
// ❌ Missing shades
primary: {
  DEFAULT: '99, 102, 241',
  '500': '99, 102, 241',
  'text': '255, 255, 255',
}

// ✅ Complete palette
primary: {
  DEFAULT: '99, 102, 241',
  '100': '224, 231, 255',
  '200': '199, 210, 254',
  // ... all shades 300-900
  'text': '255, 255, 255',
}
```

### Border Radius Inconsistency

**Match Your Design System:**
```typescript
// ❌ Don't mix border radius styles
borderRadius: 'none',
cardBackgroundStyle: {
  borderRadius: '16px',  // Conflicts with theme setting
}

// ✅ Use consistent border radius
borderRadius: 'large',
// cardBackgroundStyle respects theme borderRadius
```

### Hiding Critical Elements

**Consider User Needs:**
```typescript
// ⚠️  Be careful hiding all header elements
header: {
  hideMenu: true,
  hideTabs: true,
  hideWallets: true,
}
// Users may need these for navigation and status
```

Only hide elements when you have alternative UI or when they're truly not needed for your use case.

## Layout and Spacing

### Transparent Backgrounds

**When Using Transparency:**
```typescript
cardBackgroundStyle: {
  backgroundColor: 'transparent',
}
```

**Ensure:**
- Your app background provides sufficient contrast
- Text remains readable
- Borders are visible if needed
- Interactive elements are distinguishable

### Custom Padding

**Be Consistent:**
```typescript
// Good: Consistent spacing
cardBackgroundStyle: {
  padding: '24px',
}

// Avoid: Uneven spacing
cardBackgroundStyle: {
  padding: '10px 50px 5px 30px',
}
```

## Branding

### White Labeling

**When Hiding "Powered By":**
```typescript
hidePoweredBy: true
```

**Considerations:**
- Verify your agreement allows white labeling
- Consider brand attribution requirements
- Maintain trust with transparency
- Provide alternative attribution if needed

### Header Visibility

**Strategic Header Configuration:**
```typescript
// Minimal embedded widget
header: {
  hideMenu: true,
  hideTabs: true,
  hideWallets: true,
}

// Full-featured standalone widget
header: {
  hideMenu: false,
  hideTabs: false,
  hideWallets: false,
}
```

## Version Control

### Organize Your Themes

**File Structure:**
```
/themes
  ├── darkTheme.ts
  ├── lightTheme.ts
  ├── brandTheme.ts
  └── index.ts
```

**Export Pattern:**
```typescript
// themes/index.ts
export { darkTheme } from './darkTheme';
export { lightTheme } from './lightTheme';
export { brandTheme } from './brandTheme';
```

### Document Your Choices

**Add Comments:**
```typescript
const customTheme = {
  // Brand primary: Indigo 500
  primary: {
    DEFAULT: '99, 102, 241',
    // Generated from https://tailwindshades.app using #6366f1
    '100': '224, 231, 255',
    // ...
  },

  // Glassmorphism effect for modern look
  cardBackgroundStyle: {
    backgroundColor: 'rgba(255, 255, 255, 0.05)',
    backdropFilter: 'blur(20px)',
  },
};
```

## Checklist

Before deploying your customized widget, verify:

**Colors:**
- [ ] All colors use correct RGB format (without `rgb()`)
- [ ] Complete color palettes (100-900 shades)
- [ ] Text colors have sufficient contrast
- [ ] Status colors follow semantic meaning
- [ ] Colors tested with color blindness simulator

**Configuration:**
- [ ] Border radius matches design system
- [ ] Header visibility suits use case
- [ ] Card background works on app background
- [ ] Custom styling doesn't conflict with theme

**Testing:**
- [ ] Tested in playground
- [ ] Works on mobile devices
- [ ] Performs well on low-end devices
- [ ] Compatible across browsers
- [ ] Verified on actual application background

**Accessibility:**
- [ ] Meets WCAG 2.1 contrast requirements
- [ ] Text is readable in all states
- [ ] Status indicators use more than just color
- [ ] Interactive elements are distinguishable

**Performance:**
- [ ] No excessive blur effects
- [ ] Simple gradients
- [ ] Reasonable shadow effects
- [ ] Fast load times
